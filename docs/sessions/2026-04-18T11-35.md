## 1. Goals

- Explain how a CLI should handle the edge case where a user pipes tabular output into a pager but non-TTY detection would otherwise switch output to JSON
- Decide whether an environment variable should be allowed to set a default output format
- If yes, add a spec requirement describing the env var contract and precedence

## 2. Work Done

- Reviewed existing challenges and requirements related to non-TTY detection, pager suppression, output modes, and pipeline behavior
- Confirmed the current spec treats `stdout` non-TTY as a trigger for JSON output by default
- Recommended keeping `--output` as the canonical interface and using an explicit human-readable mode such as `--output table` for pager pipelines
- Evaluated whether a generic env var such as `_OUTPUT` should be supported and recommended a tool-scoped variable instead
- Added [requirements/o-042-output-format-env-var-default.md](requirements/o-042-output-format-env-var-default.md)
- Updated [requirements/o-001-output-format-flag.md](requirements/o-001-output-format-flag.md) to document that the new env var requirement provides only a default and never overrides `--output`
- Updated [requirements/index.md](requirements/index.md) to include `REQ-O-042` and increment the requirement counts
- Ran requirement/schema symmetry checks and index completeness checks
- Ran focused link validation for the changed files and their new cross-references

## 3. Decisions Made

| Decision | Rationale |
|----------|-----------|
| Keep non-TTY auto-selection behavior unchanged for bare `tool list \| more` | The existing spec prioritizes predictable machine-readable output in non-TTY contexts |
| Treat `--output` as the authoritative output-selection interface | It keeps invocation semantics explicit and consistent across tools |
| Add support for a tool-scoped env var default such as `TOOL_OUTPUT` | It solves the human shell-session use case without weakening explicit CLI flags |
| Define env-var precedence as `--output` first, then `<TOOLNAME>_OUTPUT`, then normal auto-detection | This preserves explicit user intent and keeps existing non-TTY behavior intact when no override is set |
| Reject generic env vars like `_OUTPUT` and `OUTPUT` | They are ambiguous, collision-prone, and inconsistent with the repo's tool-scoped env var patterns |
| Model the new behavior as an opt-in requirement `REQ-O-042` | It fits the existing framework of optional global features and does not require new wire-format fields |

## 4. Decisions Refused / Alternatives Rejected

| Rejected Approach | Reason |
|-------------------|--------|
| Supporting `_OUTPUT=table` as the recommended contract | The name is too generic and does not fit the established tool-scoped environment variable pattern |
| Inferring that a downstream pager means the CLI should silently switch back to table output | That behavior would be fragile, platform-specific, and would undermine predictable non-TTY JSON behavior |
| Letting an environment variable override an explicit `--output` flag | Explicit invocation should always win for reproducibility and clarity |
