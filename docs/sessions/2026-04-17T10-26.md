## 3. Decisions Made

| Decision | Rationale |
|----------|-----------|
| `--output` remains the sole canonical flag for response representation | The requirement text already standardized on `--output`, and a single name improves cross-command and cross-tool consistency |
| The spec should discourage aliases such as `--format` | Aliases add discovery ambiguity, complicate help and schema extraction, and weaken transferability of agent behavior |
| The rationale should not claim exclusive LLM training on `--output` | That claim is too strong; consistency and interoperability are stronger and more defensible reasons |
| The conflicting help example should be normalized to `--output` | Examples should match the normative requirement language to avoid drift inside the spec |

## 4. Decisions Refused / Alternatives Rejected

| Rejected Approach | Reason |
|-------------------|--------|
| Require `--format` as an alias for `REQ-O-001` | The current requirement does not require it, and adding it would introduce ambiguity without solving a spec gap |
| Justify the design by saying LLMs are trained exclusively for `--output` | That overstates what can be supported and risks making the spec rationale unsound |
